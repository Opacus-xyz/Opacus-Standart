## Abstract

[ERC-8201](./eip-8201.md) specifies an on-chain registry for AI agents indexed by Uber's H3 hexagonal geospatial grid. Agents register by staking a bond in ETH or [ERC-20](./eip-20.md) tokens; a configurable fee is collected at registration time and forwarded to a fee recipient determined by the implementation. Consumers discover agents by capability and geographic proximity, enabling latency-aware, physical-world-aware multi-agent coordination.

## Motivation

As autonomous agents proliferate across decentralised physical infrastructure (DePIN), robotics, and IoT, routing decisions increasingly depend on geographic proximity. H3 is a widely adopted, deterministic hexagonal grid system that encodes locations as compact 64-bit integer indexes. By anchoring agent records to H3 cells on-chain, ERC-8201 enables:

* **Latency-aware routing** — peers select the geographically nearest agent.
* **Coverage proofs** — H3 cells provide verifiable area coverage for SLA enforcement.
* **Staking alignment** — bonded fees create Sybil resistance and protocol alignment.

No existing ERC defines an application-layer interface for geospatial indexing of agent endpoints. This standard fills that gap.

## Specification

### Constants

| Name | Value | Description |
|---|---|---|
| `H3_FEE_BPS` | `100` | Default registration fee in basis points (1 %) |

> **Note:** The address that receives the registration fee (`feeRecipient`) is implementation-defined. Implementations MUST make `feeRecipient` configurable (e.g., via constructor parameter or governance-controlled setter) and MUST NOT hardcode a chain-specific address.

### Structs

```solidity
struct AgentRecord {
    address   wallet;
    bytes32   h3Index;         // H3 cell (64-bit index zero-padded to 32 bytes)
    string    quicEndpoint;    // "ip:port" or multiaddr
    bytes32[] capabilities;    // keccak256-hashed capability labels
    uint256   kineticScore;    // ERC-7755 reputation score (0–10 000)
    address   token;           // registration bond token (address(0) = ETH)
    uint256   regFeeGross;     // original gross bond
    uint64    registeredAt;
    uint64    expiresAt;
    bool      active;
}
```

### Interface

```solidity
interface IERC8201 {
    function registerAgent(
        bytes32 h3Index,
        string calldata quicEndpoint,
        bytes32[] calldata capabilities,
        address token,
        uint256 gross
    ) external payable returns (bytes32 did);

    function updateAgent(
        bytes32 h3Index,
        string calldata quicEndpoint,
        bytes32[] calldata capabilities
    ) external;

    function deregisterAgent() external;

    function discoverAgents(
        bytes32 capability,
        bytes32 h3Near,
        uint256 minScore,
        uint256 limit
    ) external view returns (AgentRecord[] memory agents);
}
```

### Methods

#### `registerAgent`

```solidity
function registerAgent(
    bytes32 h3Index,
    string calldata quicEndpoint,
    bytes32[] calldata capabilities,
    address token,
    uint256 gross
) external payable returns (bytes32 did);
```

The `registerAgent` function handles both native ETH and [ERC-20](./eip-20.md) deposits, distinguished by `token`:

- `token == address(0)` **MUST** indicate a native ETH deposit. `msg.value` **MUST** equal `gross`. MUST NOT perform any [ERC-20](./eip-20.md) calls.
- `token != address(0)` **MUST** indicate an [ERC-20](./eip-20.md) deposit. `msg.value` **MUST** equal zero. MUST pull funds via `IERC20(token).transferFrom(msg.sender, address(this), gross)` and MUST revert on failure.

In both cases:

- **MUST** compute `feeAmount = (gross + 99) / 100` (1 % rounded up using integer division) and forward `feeAmount` to `feeRecipient` atomically.
- **MUST** compute `bondAmount = gross - feeAmount` and lock `bondAmount` as a recoverable bond.
- **MUST** set `registeredAt = block.timestamp` (UNIX seconds from `block.timestamp`; MUST NOT change after registration).
- **MUST** return a deterministic DID: `keccak256(abi.encodePacked(h3Index, msg.sender))`.
- **MUST** revert if the caller is already registered.
- **MUST** emit `AgentRegistered(wallet, h3Index, quicEndpoint, feeAmount)`.

#### `updateAgent`

```solidity
function updateAgent(
    bytes32 h3Index,
    string calldata quicEndpoint,
    bytes32[] calldata capabilities
) external;
```

MUST only be callable by the registered agent (`msg.sender == wallet`).  
MUST emit `AgentUpdated(wallet, h3Index)`.

#### `deregisterAgent`

```solidity
function deregisterAgent() external;
```

MUST release `netAmount` (99 % of original bond) back to `msg.sender`.  
MUST emit `AgentDeregistered(wallet)`.

#### `discoverAgents`

```solidity
function discoverAgents(
    bytes32 capability,
    bytes32 h3Near,
    uint256 minScore,
    uint256 limit
) external view returns (AgentRecord[] memory agents);
```

MUST filter by `capability` membership and `kineticScore >= minScore`.  
`h3Near` proximity filtering MAY be performed off-chain in V1.

### Events

```solidity
event AgentRegistered(address indexed wallet, bytes32 h3Index, string quicEndpoint, uint256 feeAmount);
event AgentUpdated(address indexed wallet, bytes32 h3Index);
event AgentDeregistered(address indexed wallet);
```

### [ERC-165](./eip-165.md)

Compliant contracts MUST implement the `IERC8201` interface defined by this standard and MUST return `true` from `supportsInterface` for `type(IERC8201).interfaceId`.

## Rationale

* **H3 as bytes32** – The 64-bit H3 index fits in `bytes32`, allowing compact on-chain storage and future extension.
* **Bond-not-stake** – The registration bond is fully refundable (minus the 1 % fee), avoiding liquidity lock risk.
* **Capability as keccak256** – Fixed-size capability identifiers enable O(1) lookup in Solidity mappings.

## Security Considerations

* **Sybil resistance** – The bonded fee creates economic cost for mass registrations.
* **Reentrancy** – `deregisterAgent` transfers ETH/[ERC-20](./eip-20.md) and MUST be protected by a reentrancy guard.
* **Endpoint validation** – `quicEndpoint` is stored as an opaque string; consumers MUST validate format off-chain.
* **H3 spoofing** – `h3Index` is self-reported and cannot be verified on-chain; consumers SHOULD probe physical connectivity before routing latency-sensitive tasks.
* **Fee recipient** – Implementations MUST protect `feeRecipient` mutation with appropriate access control (e.g., owner-only or governance-controlled).
* **ERC-20 allowance** – `registerAgent` calls `transferFrom`; agents MUST approve the registry before calling and MUST NOT leave excess allowance.
