## Abstract

[ERC-8202](./eip-8202.md) defines a proof-based escrow protocol for autonomous agent task markets. A requester deposits funds (ETH or [ERC-20](./eip-20.md) tokens) into an escrow contract; funds are released to the provider only after a cryptographic proof of task completion is verified on-chain. A configurable dispute window and arbiter address provide recourse for contested proofs. ERC-8202 extends V1 escrow semantics with partial release, multi-proof batching, and resolver hook support.

## Motivation

Existing escrow standards release funds upon either (a) manual approval by the requester or (b) a timeout. Neither guarantees objective verification. As autonomous agents exchange services for payment, subjective release criteria introduce asymmetric trust. ERC-8202 replaces human approval with verifiable proofs — zero-knowledge receipts, signed oracle attestations, or IPFS-anchored work logs — so fund release becomes permissionless once proof validity is established.

Compared to V1 escrow implementations:
* **Partial release** — emit progress proofs and unlock proportional tranches.
* **Batch proofs** — verify multiple task receipts in one transaction.
* **Resolver hooks** — plug-in arbitrary verifier contracts without changing escrow logic.

## Specification

### Constants

| Name | Value | Description |
|---|---|---|
| `DISPUTE_WINDOW` | `172 800` | Default dispute window in seconds (48 hours) |
| `MAX_FEE_BPS` | `500` | Maximum platform fee in basis points (5 %) |

> **Note:** The address that receives the platform fee (`feeRecipient`) is implementation-defined. Implementations MUST make `feeRecipient` configurable and MUST NOT hardcode a chain-specific address.

### Structs

```solidity
struct EscrowRecord {
    address   requester;
    address   provider;
    address   token;            // address(0) = ETH
    uint256   totalAmount;      // gross deposit
    uint256   releasedAmount;   // cumulative released so far
    address   resolver;         // proof verifier contract
    address   arbiter;          // dispute arbiter address
    uint64    createdAt;
    uint64    disputeDeadline;  // createdAt + DISPUTE_WINDOW
    bytes32   jobHash;          // keccak256 of off-chain job spec
    uint8     status;           // 0=Open, 1=Completed, 2=Disputed, 3=Cancelled
}
```

### Interface

```solidity
interface IERC8202 {
    function createEscrow(
        address provider,
        address token,
        uint256 amount,
        address resolver,
        address arbiter,
        bytes32 jobHash
    ) external payable returns (bytes32 escrowId);

    function submitProof(
        bytes32 escrowId,
        bytes calldata proof,
        uint256 releaseAmount
    ) external;

    function batchSubmitProofs(
        bytes32[] calldata escrowIds,
        bytes[]   calldata proofs,
        uint256[] calldata releaseAmounts
    ) external;

    function raiseDispute(bytes32 escrowId) external;

    function resolveDispute(
        bytes32 escrowId,
        address winner,
        uint256 providerAmount
    ) external;

    function cancelEscrow(bytes32 escrowId) external;

    function getEscrow(bytes32 escrowId) external view returns (EscrowRecord memory);
}
```

### Methods

#### `createEscrow`

```solidity
function createEscrow(
    address provider,
    address token,
    uint256 amount,
    address resolver,
    address arbiter,
    bytes32 jobHash
) external payable returns (bytes32 escrowId);
```

- `token == address(0)` MUST indicate a native ETH deposit. `msg.value` MUST equal `amount`.
- `token != address(0)` MUST indicate an [ERC-20](./eip-20.md) deposit. `msg.value` MUST equal zero. MUST call `IERC20(token).transferFrom(msg.sender, address(this), amount)` and revert on failure.
- MUST derive `escrowId = keccak256(abi.encodePacked(msg.sender, provider, jobHash, block.timestamp))`.
- MUST set `disputeDeadline = block.timestamp + DISPUTE_WINDOW`.
- MUST emit `EscrowCreated(escrowId, requester, provider, amount)`.

#### `submitProof`

```solidity
function submitProof(
    bytes32 escrowId,
    bytes calldata proof,
    uint256 releaseAmount
) external;
```

- MUST only be callable by `provider`.
- MUST call `IProofResolver(resolver).verify(escrowId, proof)` and revert if it returns `false`.
- `releaseAmount` MUST be ≤ `totalAmount - releasedAmount`.
- MUST transfer `releaseAmount` minus fee to `provider` and fee to `feeRecipient`.
- If `releasedAmount + releaseAmount == totalAmount` MUST set `status = 1` (Completed).
- MUST emit `ProofSubmitted(escrowId, proof, releaseAmount)`.

#### `batchSubmitProofs`

```solidity
function batchSubmitProofs(
    bytes32[] calldata escrowIds,
    bytes[]   calldata proofs,
    uint256[] calldata releaseAmounts
) external;
```

MUST process each triple `(escrowId, proof, releaseAmount)` sequentially, applying the same rules as `submitProof`. All array lengths MUST be equal; reverts otherwise.

#### `raiseDispute`

```solidity
function raiseDispute(bytes32 escrowId) external;
```

- MUST only be callable by `requester` within `disputeDeadline`.
- MUST set `status = 2` (Disputed).
- MUST emit `DisputeRaised(escrowId, requester)`.

#### `resolveDispute`

```solidity
function resolveDispute(
    bytes32 escrowId,
    address winner,
    uint256 providerAmount
) external;
```

- MUST only be callable by `arbiter`.
- `providerAmount` MUST be ≤ `totalAmount`.
- Remaining funds `totalAmount - providerAmount` MUST be returned to `requester`.
- MUST set `status = 1` (Completed).
- MUST emit `DisputeResolved(escrowId, winner, providerAmount)`.

#### `cancelEscrow`

```solidity
function cancelEscrow(bytes32 escrowId) external;
```

- MUST only be callable by `requester` if `status == 0` (Open) and no proof has been submitted.
- MUST return full `totalAmount` to `requester`.
- MUST set `status = 3` (Cancelled).
- MUST emit `EscrowCancelled(escrowId)`.

### Events

```solidity
event EscrowCreated(bytes32 indexed escrowId, address indexed requester, address indexed provider, uint256 amount);
event ProofSubmitted(bytes32 indexed escrowId, bytes proof, uint256 releaseAmount);
event DisputeRaised(bytes32 indexed escrowId, address requester);
event DisputeResolved(bytes32 indexed escrowId, address winner, uint256 providerAmount);
event EscrowCancelled(bytes32 indexed escrowId);
```

### Resolver Hook Interface

```solidity
interface IProofResolver {
    function verify(bytes32 escrowId, bytes calldata proof) external view returns (bool);
}
```

Implementations MAY provide built-in resolvers for ECDSA attestations, EIP-712 signed receipts, and IPFS hash anchors.

### [ERC-165](./eip-165.md)

Compliant contracts MUST implement the `IERC8202` interface and MUST return `true` from `supportsInterface` for `type(IERC8202).interfaceId`.

## Rationale

* **Resolver hooks** — Separating proof verification into a plug-in contract allows upgrading verification logic without migrating escrow state.
* **Partial release** — Enables milestone-based payments for long-running agent tasks, reducing lock-up risk.
* **Batch proofs** — Reduces gas cost for high-frequency micro-task markets where many proofs arrive in the same block.
* **Dispute window** — A fixed window (default 48 h) balances requester recourse against provider payment latency.

## Backwards Compatibility

No existing ERC is modified. V1 escrow implementations remain unaffected.

## Security Considerations

* **Reentrancy** — All state changes MUST occur before external token transfers. Use a reentrancy guard on `submitProof`, `resolveDispute`, and `cancelEscrow`.
* **Proof replay** — Resolvers MUST track consumed proof nonces or use escrow-scoped commitment schemes to prevent double-release.
* **Arbiter trust** — The `arbiter` address has unilateral power to redistribute funds; requesters SHOULD only accept trusted or governance-controlled arbiters.
* **[ERC-20](./eip-20.md) allowance** — Providers MUST approve the escrow contract before `createEscrow` and MUST NOT leave excess allowance.
* **Fee cap** — Implementations MUST enforce `feeBps <= MAX_FEE_BPS` to protect providers from excessive extraction.
* **Overflow** — Use checked arithmetic; `releasedAmount + releaseAmount` MUST NOT overflow `totalAmount`.
